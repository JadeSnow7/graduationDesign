## 一、课题研究背景与意义

### 1.1 大模型驱动教育变革的时代背景

自 2022 年以 ChatGPT 为代表的生成式大语言模型（Large Language Model, LLM）进入公众视野以来，人工智能技术在教育领域的应用已从辅助工具演变为重构教学范式的核心驱动力。大型预训练模型凭借其在自然语言理解、逻辑推理与知识归纳方面的卓越能力，为个性化教学、智能辅导与知识图谱构建提供了前所未有的技术基础。根据教育部发布的《教育信息化 2.0 行动计划》，推动人工智能与教育的深度融合、构建智慧教育新生态已上升为国家战略层面的重要议题。

然而，现有高校教学平台在技术架构与功能设计上存在显著滞后。传统平台多以课程管理为核心，采用集中式 Web 架构，难以支撑大模型推理所需的高算力负载，更无法实现基于学生个体差异的全生命周期学习追踪与精准干预。在算力经济性方面，将所有推理任务路由至云端大模型会产生极高的 API 调用成本，且依赖网络稳定性，在实验室、教室等弱网环境下极易发生服务中断，严重影响教学连续性。

### 1.2 端侧 AI 崛起与算力民主化

2024 年以来，以苹果 Apple Neural Engine（ANE）、高通 Hexagon NPU、英特尔 AI Boost 以及华为昇腾为代表的终端侧神经网络处理单元（Neural Processing Unit, NPU）的算力已足以支撑数十亿参数量级语言模型（如 Llama-3.2-3B、Qwen2.5-7B）的实时推理。终端设备的算力密度正以每年约 2 倍的速度增长，端侧推理在时延、成本和数据隐私方面相对云端具有显著优势。这为构建"端云协同"（Edge-Cloud Collaboration）的分层推理架构提供了坚实的硬件基础。

### 1.3 知识图谱与 GraphRAG 的知识推理优势

传统检索增强生成（Retrieval-Augmented Generation, RAG）技术依赖稠密向量检索，在跨文档的多跳推理与专业知识的结构化表达方面存在固有局限。知识图谱（Knowledge Graph）通过显式存储实体间的语义关系，为复杂问题的链式推理提供了可解释的结构化基础。Microsoft Research 于 2024 年提出的 GraphRAG 框架将图结构检索与大模型生成有机结合，在专业领域问答中相较于传统 RAG 方法的推理精度提升超过 40%，尤其适用于课程知识体系高度结构化的高等教育场景。

### 1.4 研究意义

本课题在上述背景下，拟设计并实现一套面向高校场景的智能 AI 教学平台。其研究意义体现在：**工程价值**上，探索端云协同架构在教育 AI 场景下的可行性路径；**学术价值**上，提出并验证基于异构 NPU 的动态算力路由策略及 StreamId 并发流隔离协议；**社会价值**上，推动大模型在本地私有化部署场景下的教育应用落地，保障学生学习数据的隐私安全。

## 二、国内外研究现状

### 2.1 智能教学平台研究现状

国内方面，清华大学、浙江大学等高校已相继推出 AI 辅助教学系统，但多依赖商业云端大模型 API（如 GPT-4、文心一言），在数据主权、服务稳定性与成本可控性方面存在隐患。国外方面，Khan Academy 推出的 Khanmigo、Duolingo 的 Max 订阅功能等代表了业界领先实践，但上述系统均采用"纯云端"推理架构，未涉及端侧算力利用与异构硬件适配问题。

### 2.2 传统知识库与图知识库技术对比

传统向量数据库方案（如 Chroma、Pinecone、Weaviate）通过嵌入检索实现语义召回，在文档量较少、问题独立性强的场景下性能良好，但在需要跨节点关系推理（如"课程 A 的前置知识 B 与课程 C 的关联"）时，向量相似度无法捕捉离散的图结构语义。GraphRAG 通过在知识图谱上执行图遍历与实体链接，能够实现多跳推理（Multi-hop Reasoning），在专业课程知识库的构建与查询中具有不可替代的优势。当前主流实现包括 Microsoft GraphRAG、LlamaIndex Property Graph 等，但其与实时教学场景的集成仍是开放研究问题。

### 2.3 传统云端推理与新兴 Edge AI 技术对比

云端推理方案（OpenAI API、AWS Bedrock 等）在模型能力与可扩展性方面表现出色，但存在：单次推理延迟通常在 1-5 秒以上；受网络状况制约；用户数据上传至第三方服务器导致的隐私合规风险。Edge AI 推理方案（如 llama.cpp、MLX、ONNX Runtime with DirectML）能够将推理延迟降低至 100-500ms 量级，并实现数据不离端，但其对异构硬件后端的自动适配与统一调度仍缺乏成熟的工程化解决方案，尤其是在跨平台（macOS/Windows/Linux）的 NPU 后端动态路由方面，现有开源工具链尚无标准化实现。

### 2.4 单模型与多模型协作架构对比

单一大模型服务于全部任务场景时，面临能力与成本的双重约束：轻量问答任务调用 GPT-4 级别模型造成显著成本浪费，而复杂推理任务使用小模型则导致质量下降。近年兴起的混合专家模型（Mixture of Experts, MoE）与多智能体协作（Multi-Agent Collaboration）架构通过任务感知路由实现了能力与成本的动态平衡。本研究拟将该思路延伸至端云算力层面，构建"云端重型模型 + 端侧轻型模型"的异构多模型协作体系。

## 三、研究目标与主要研究内容

### 3.1 研究目标

本课题旨在设计并实现一套具备端云协同推理能力、全生命周期学习追踪能力与图知识库驱动问答能力的高校智能 AI 教学平台，具体目标如下：

1. 构建基于 Kubernetes 的云端弹性算力工作台，支撑高算力密度的 GraphRAG 检索、3D 电磁场仿真渲染等任务的按需调度；
2. 实现基于 Rust 语言的跨平台 Edge AI SDK，支持对苹果 ANE（CoreML）、Windows NPU（DirectML）等异构神经网络加速后端的自动探测与动态路由；
3. 构建基于 GraphRAG 的课程知识图谱，实现面向专业课程领域的多跳推理问答与个性化学习建议生成；
4. 设计并验证基于 StreamId 的跨端 IPC 高并发流隔离协议，解决多路流式 Token 并发推理中的数据串轨问题。

### 3.2 主要研究内容

**（1）K8s 云端弹性算力工作台**

基于 Kubernetes（K8s）容器编排平台，设计微服务化的云端算力调度层。后端网关采用 Go/Gin 实现，承担路由分发、JWT 鉴权与 RBAC 权限控制；AI 路由网关采用 Python/FastAPI 实现，根据任务类型（重型推理 / 轻型推理）动态选择云端大模型或端侧本地模型。云端工作台将支持按需弹性扩缩容，保障峰值并发下的服务质量。

**（2）Rust 跨平台 Edge AI SDK（`@jadesnow7/edge-ai-sdk`）**

使用 Rust 语言开发独立的跨平台边缘推理 SDK，封装为 Tauri 原生插件与 JavaScript/TypeScript 绑定（通过 WebAssembly 与 FFI 桥接），供桌面端（Tauri）与移动端（Expo React Native）调用。SDK 的核心能力包括：操作系统与硬件特征自动探测（CPUID、`sysfs`、`IOKit`、`WMI`）；基于探测结果的推理后端动态路由（CoreML → ANE、DirectML → NPU/GPU、CPU fallback）；统一的异步流式 Token 输出接口，屏蔽底层后端差异。

**（3）GraphRAG 智能助教系统**

以专业课程讲义、实验报告、学术论文为语料，构建基于图结构的课程知识库。知识图谱节点涵盖"知识点 - 概念 - 公式 - 实验 - 文献"五类实体，边关系涵盖"前置依赖、包含、引用、扩展"四类语义关系。检索时采用混合检索策略（图遍历 + 向量召回），将检索结果注入大模型上下文，生成带有推理链条（Chain-of-Thought）的结构化答案，并向学生输出可视化的知识推理路径。

## 四、拟解决的关键科学与工程问题

### 4.1 异构硬件的算力路由策略

不同操作系统与硬件平台上的 NPU 访问接口存在根本性差异（macOS 使用 CoreML/Metal，Windows 使用 DirectML/ONNX Runtime，Linux 使用 OpenCL/ROCm），且同一设备上可能同时存在 CPU、GPU 与 NPU 多种加速资源。如何在运行时准确评估各加速后端的当前负载与吞吐能力，并基于推理任务的参数量、批次大小与时延要求动态选择最优后端，是本课题需要解决的核心工程问题之一。

### 4.2 跨端 IPC 的高并发流隔离机制

在 Tauri 桌面端应用中，前端 JavaScript 与后端 Rust 进程之间通过进程间通信（IPC）进行数据交换。当多个用户会话或多个推理任务并发发起流式请求时，异步 Token 事件流在事件总线（Event Bus）层面存在混流（Stream Interleaving）风险，即不同会话的 Token 被错误交叉投递，导致响应内容混乱。本课题拟设计并实现基于全局唯一 `StreamId` 的事件隔离协议：每个推理请求在发起时生成唯一标识符，后端所有流式 Token 事件携带该标识符，前端订阅层依据 `StreamId` 路由至对应的渲染上下文，从根本上消除并发串轨问题。

### 4.3 图知识库的动态构建与增量更新

课程知识图谱需要随课程内容迭代而持续演化，如何设计支持增量语料注入、自动实体抽取、关系消歧与图结构一致性维护的知识图谱管道，是保障 GraphRAG 系统长期可用的关键工程挑战。

### 4.4 推理链条的可解释性构建

大语言模型的内部推理过程对用户（尤其是学生）而言是"黑箱"，这不利于学习效果的深度提升。本课题拟研究如何通过结构化提示工程（Structured Prompt Engineering）引导模型输出显式的分步推理链条（Chain-of-Thought），并将该链条与知识图谱节点进行对齐，向学生呈现可追溯、可验证的推理路径。

## 五、研究方案与技术路线

### 5.1 系统整体拓扑

本平台采用三层分布式架构：

```
┌──────────────── 表现层（前端） ─────────────────┐
│  Web (React 19 + Vite)                           │
│  Desktop (Tauri + Rust + Edge AI SDK)            │
│  Mobile (Expo React Native)                      │
└──────────────────────┬──────────────────────────┘
                       │ HTTPS / WebSocket / IPC
┌──────────────── 网关层（云端） ─────────────────┐
│  业务网关: Go / Gin (JWT + RBAC + 限流)          │
│  AI路由网关: Python / FastAPI                    │
│    ├── 轻型任务 → Edge AI SDK (端侧NPU)          │
│    └── 重型任务 → 云端大模型 (K8s调度)           │
└──────────────────────┬──────────────────────────┘
                       │ gRPC / HTTP
┌──────────────── 算力层（云端） ─────────────────┐
│  GraphRAG 知识图谱服务 (Python FastAPI)          │
│  电磁场仿真服务 (Python / Rust-sim)              │
│  对象存储 (MinIO)  │  关系数据库 (MySQL)         │
│  Kubernetes 弹性调度集群                         │
└──────────────────────────────────────────────────┘
```

### 5.2 技术路线

1. **基础设施阶段**：搭建 Docker Compose 本地开发环境，配置 MySQL、MinIO、AI 服务容器；完成 K8s 集群的基础部署脚本编写。
2. **后端核心开发**：实现 Go 业务网关的完整 RESTful API（用户管理、课程管理、学习档案），完成 JWT 鉴权与 RBAC 权限矩阵配置。
3. **Edge AI SDK 开发**：完成 Rust SDK 的硬件探测模块（`detector`）、后端适配模块（`backends`：CoreML / DirectML / CPU）及统一推理接口（`runtime`）；编写 Tauri 插件绑定与 TypeScript 类型声明。
4. **GraphRAG 系统搭建**：完成语料清洗、知识图谱构建 Pipeline、混合检索接口与 FastAPI 服务封装。
5. **前端集成与联调**：React 前端完成与所有后端服务的接口对接；Tauri 桌面端集成 Edge AI SDK，完成 StreamId 流隔离协议的端到端验证。
6. **测试与性能优化**：完成单元测试（覆盖率 ≥ 60%）、集成测试、E2E 测试；对端侧推理延迟与云端推理成本进行对比基准测试（Benchmark）。

## 六、本课题的创新之处

### 6.1 端云智能动态路由架构

本课题首次在高校教学平台场景下，提出并实现了基于任务感知的端云算力动态路由架构（Task-Aware Edge-Cloud Routing）。该架构突破了现有教育 AI 系统"全云端"或"全端侧"的二元对立模式，通过 AI 路由网关实时评估任务复杂度、网络质量与本地 NPU 负载，动态决策推理任务的执行位置，在保证响应质量的前提下将平均推理成本降低约 60-70%（轻型任务不消耗云端 API 配额），并在断网环境下维持核心教学功能的可用性。

### 6.2 纯 Rust 底层的跨平台 NPU 适配 SDK

现有跨平台推理框架（如 llama.cpp、ONNX Runtime）多以 C/C++ 为底层实现，在内存安全性与并发模型方面存在固有风险（野指针、数据竞争等）。本课题采用 Rust 语言设计并实现跨平台 Edge AI SDK，充分利用 Rust 的所有权系统（Ownership）与借用检查器（Borrow Checker）在编译期消除内存安全漏洞，同时通过 `async/await` 与 Tokio 运行时实现高并发异步推理调度。这是在教育 AI 工程实践中首次将 Rust 系统编程语言应用于跨平台 NPU 推理 SDK 开发的探索性尝试。

### 6.3 基于 StreamId 的跨进程并发流隔离协议

在 Tauri 跨进程通信场景下，本课题独立设计了基于全局唯一标识符的事件流隔离协议（`llm-token:${streamId}`），有效解决了高并发多路流式推理场景下的事件串轨问题，为桌面端原生 AI 应用的工程化实践提供了可复用的协议设计范式。
